# Performance Coach CLI 🚀

A terminal-based AI coach that leverages **NVIDIA NIM (Qwen 2.5 72B/122B)** to analyze software engineering tasks and generate high-impact, personalized completion playbooks. 

The tool maps your tasks against a specialized 9-dimension engineering rubric and provides persistent, markdown-based guidance to help you hit "Above Average" performance consistently.

## 🌟 Why Performance Coach CLI?

Most engineering tasks are defined by *what* needs to be built, but rarely *how* to build it to a high standard. This tool fills that gap by:
1. **Analyzing** the technical and soft-skill dimensions of your task.
2. **Generating** a persistent markdown playbook tailored to your specific work.
3. **Coaching** you with specific behaviors that differentiate "Average" from "Outstanding" performance.

## 📸 Full Example: From Task to Playbook

### 1. The Command
Run the coach with a complex, senior-level task:
```bash
npm run dev -- "Migrate Legacy Auth to OpenID Connect" "Replace our internal JWT-based auth with a standardized OIDC flow using Auth0. Must maintain backward compatibility for existing mobile tokens and migrate RBAC scopes to the OIDC provider."
```

## 🛠 Implementation Strategy

### 🎯 DESIGN IMPLEMENTATION

#### ✅ Above Average Behaviors
- Architects a dual-path authentication middleware that transparently routes legacy JWT validation to the internal verifier while routing OIDC tokens to Auth0.
- Creates a reusable 'ScopeMapper' utility that centrally defines the translation logic between legacy RBAC strings and OIDC scopes.
- Implements a 'TokenHydrator' pattern that automatically enriches legacy tokens with missing OIDC claims during the transition.

#### ★ Outstanding Stretch Goals
- Designs a feature-flagged 'canary' mode that allows routing a specific percentage of mobile traffic to the new OIDC flow.
- Proactively builds a simulation harness that generates malformed legacy tokens to prove the system's resilience.

#### ⚠ Pitfalls to Avoid
- Attempting a 'big bang' replacement where the old JWT verification logic is deleted before the new OIDC flow is validated.
- Hardcoding Auth0 tenant IDs and scope mappings directly into service logic instead of using configuration files.

---

### 🎯 REQUIREMENT ANALYSIS

#### ✅ Above Average Behaviors
- Documents a detailed 'Legacy Token Inventory' that lists every mobile app version currently in use.
- Explicitly validates the RBAC scope mapping with the security team, confirming exact equivalence.
- Identifies and questions the 'backward compatibility' requirement's duration, proposing a sunset timeline.

#### 💡 Coaching Nudge
- Write down three specific assumptions you are making about the mobile client behavior and ask a mobile engineer to validate them immediately.

---

### 🎯 BUILD TEST DEPLOYMENT

#### ✅ Above Average Behaviors
- Develops a comprehensive integration test suite that mocks Auth0 responses to verify handling of network timeouts.
- Configures the CI/CD pipeline to automatically spin up a staging environment with a mirrored Auth0 tenant.
- Implements a 'rollback switch' that instantly reverts traffic to the legacy JWT verifier if error rates spike.

#### 💡 Coaching Nudge
- Run the full integration test suite against the staging Auth0 instance locally before pushing code.

## 📋 Checklists

### 1️⃣ Planning (Before Starting)
- [ ] Confirm the exact list of mobile app versions in production and their token formats.
- [ ] Validate the RBAC scope mapping matrix with the security team.
- [ ] Define the 'Dual-Read' strategy and set the specific metrics for success.

### 2️⃣ Execution (While Implementing)
- [ ] Implement a feature flag to toggle between legacy and OIDC validation paths.
- [ ] Write integration tests that simulate both valid and invalid tokens.
- [ ] Update the CI/CD pipeline to include a specific job that validates the Auth0 connection.

### 3️⃣ Pre-PR (Before Review)
- [ ] All unit and integration tests pass locally and in the CI pipeline.
- [ ] The migration logic is covered by end-to-end tests in staging.
- [ ] A rollback plan is documented and attached to the PR for the review team.

## 🚀 Growth Nudges

- **When you first open the Auth0 documentation**: Don't just read; ask: 'How does this specific configuration impact our existing mobile clients?'
- **When you feel tempted to merge the code quickly**: Pause. Remember that 'Done' means running in production without incidents.

## 📖 Mentor Guide

To achieve an Above Average outcome, treat this not as a simple code replacement but as a complex system evolution. Start by deeply understanding the 'why' behind the mobile token constraints. Your design should explicitly support a 'dual-path' architecture where legacy and new tokens coexist seamlessly. Do not rush into coding; instead, spend your initial time mapping every legacy claim to its OIDC equivalent. Your implementation should prioritize readability and safety: use a centralized ScopeMapper utility and feature flags for traffic control. Crucially, your definition of success extends beyond the code merge. You must own the deployment strategy and have a clear, tested rollback plan ready.

## 🚀 Quick Start

### 1. Prerequisites
- Node.js v18+ 
- An NVIDIA NIM API Key ([Get one here](https://build.nvidia.com/explore/discover))

### 2. Installation
```bash
git clone https://github.com/madx-47/performance-coach-cli.git
cd performance-coach-cli
npm install
```

### 3. Configuration
```bash
cp .env.example .env
# Set NVIDIA_NIM_API_KEY=your_key_here
```

### 4. Run the Coach
```bash
npm run dev -- "Your Task Title" "Your Task Description"
```
